# Codebase RAG — Project Description

## Overview

Codebase RAG is a developer tool that lets you point at any GitHub repository and instantly get two things: a structured summary of what the project does, and a conversational Q&A interface where you can ask questions about the codebase and get grounded, accurate answers with source references.

The system is built entirely on free-tier services and local models — no paid APIs required for core functionality.

---

## Problem It Solves

When you encounter an unfamiliar codebase — whether it's an open source library, a new team repo, or a project you haven't touched in months — understanding it takes time. You have to read through README files, trace imports, figure out what calls what, and piece together the architecture manually.

Codebase RAG automates this. Give it a GitHub URL and within minutes you can ask natural language questions like:

- *"What does this project do at a high level?"*
- *"How does authentication work?"*
- *"Which files are responsible for the database layer?"*
- *"What does the `UserService` class depend on?"*
- *"Where is the API rate limiting implemented?"*

---

## How It Works

The system runs in two stages: **ingestion** and **retrieval**.

**Ingestion** clones the repository locally, walks through all code files, and parses them at two levels. First, AST parsing (via tree-sitter) extracts the structural elements — functions, classes, imports, and call relationships — and builds a knowledge graph using NetworkX. Second, the code is split into smart chunks (by function and class boundaries, not arbitrary token windows) and embedded locally using `nomic-embed-text-v1.5`, a high-quality embedding model that runs entirely on your machine. These embeddings are stored in a local ChromaDB instance.

**Retrieval** uses a hybrid approach that combines vector similarity search with graph traversal. When you ask a question, the system finds semantically similar code chunks via ChromaDB, then expands the result set by walking the call graph to pull in related functions and files that may not have matched semantically but are structurally relevant. This two-step retrieval gives significantly better results than vector search alone, especially for questions that require cross-file reasoning.

The final answer is generated by Groq's Llama 3.1 70B (for Q&A) or Gemini 1.5 Flash (for full project summarization), both available on generous free tiers.

## Tech Stack

| Component | Technology |
|---|---|
| Repo cloning | `gitpython` |
| AST parsing | `tree-sitter` |
| Embeddings | `sentence-transformers` — `nomic-embed-text-v1.5` (local) |
| Vector store | ChromaDB (local persistent) |
| Graph store | NetworkX (in-memory, serialized to disk) |
| Q&A LLM | Groq — Llama 3.1 70B (free tier) |
| Summarization LLM | Google Gemini 1.5 Flash (free tier) |
| Backend API | FastAPI |
| Frontend | Streamlit |

## Key Design Decisions

**Why hybrid RAG over pure vector RAG?**
Code is inherently relational. A function that authenticates users may not mention "authentication" in its own body — but it's called from a route that does. Vector search alone misses these connections. The graph layer captures `imports`, `calls`, and `defines` relationships so retrieval can follow the actual structure of the code.

**Why local embeddings?**
Using `sentence-transformers` with `nomic-embed-text-v1.5` means zero embedding costs, no rate limits, and no data leaving your machine. The model performs on par with OpenAI's `text-embedding-3-small` on code retrieval benchmarks.

**Why Groq for Q&A?**
Groq's inference speed (500+ tokens/sec) makes the Q&A feel instant. At free tier limits it's more than sufficient for development and moderate usage.

**Why Gemini Flash for summarization?**
Summarizing a project requires fitting a lot of context — README, entry points, package files, directory structure — into a single prompt. Gemini Flash's 1M token context window handles even large repos comfortably, and the free tier provides 1M tokens/day.

## Scope and Limitations

- Supports Python, JavaScript, and TypeScript for AST-level parsing. Other languages fall back to text-based chunking.
- Works best on repos under ~100MB. Very large monorepos may require selective directory ingestion.
- Graph traversal is limited to static analysis — it does not execute code or resolve dynamic imports.
- Private repos require a GitHub personal access token.
- All data (embeddings, graphs, cloned code) is stored locally. No data is sent to third parties except the LLM prompt text sent to Groq/Gemini.
